Modernising software for chemical labs

Problem

In a world of labaratory equipment things are complicated: there are too many different tools with their own complicated workflows and UI; new equipment keeps getting developed and labs change their setups; users range from occasional newbies to hardcore power-users. So a big hardware producer decided to leverage their experiense and build a common software platform which could be additionally offered to their customers as a paid software.

Solution

Our team was building a multi-platform web application which should support a plug in approach for additional equipment on top of providing the basic common functionality as a default. Several things which I was working on to facilitate that are: a design system with theaming capabilities, a library of possible layouts and a comprehensive documentation.

Results

The platform's development still continues while I am no longer working on the project. The work I've done helped the team to continue uninterupted development without a designer for a while.

Learnings

  • The more complicated the domain the more valuable is the access to the actual users with interviews and prototypes testing;
  • Same with the user personas - with such functionality they are essential;
  • Working on regulated environment projects sets unexpected limitations;
  • Accessibility has to be one of the top priorities;
  • The stakeholders' decision of not catering for touch devices separately complicated things down the line;
  • Calling something an MVP while expecting a full-fledged final application does not magically make it happen on an MVP timeline.
